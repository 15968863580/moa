## 1. 项目概述

### 1.1 项目背景

Mixture of Agents (MOA) 是一种多模型协作架构，通过多个 Reference 模型并行生成回答，再由 Aggregator 模型汇总优化，显著提升输出质量。目前该功能仅在 Hermes Agent 内部实现，无法被其他系统复用。

本项目旨在将 MOA 能力独立为一个标准化服务，对外提供 OpenAI 兼容的 API 接口，使任何支持 OpenAI API 的 Agent 系统（如 Dify、FastGPT、自建系统等）都能无缝接入 MOA 能力。

### 1.2 核心目标

- **OpenAI API 兼容**：完全兼容 `/v1/chat/completions` 接口，支持流式和非流式
- **多模型协作**：支持配置多个 Reference 模型并行处理，一个 Aggregator 模型汇总
- **灵活配置**：通过配置文件定义多组 MOA 预设，不同 model name 对应不同配置
- **高性能**：并行调用 Reference 模型，最小化延迟
- **易部署**：Docker 容器化，一键启动
- **可观测**：内置日志、监控、调用追踪

### 1.3 目标用户

- 使用 Dify、FastGPT、Langchain 等 Agent 框架的开发者
- 需要提升 AI 输出质量的团队
- 希望使用多模型协作但不想自己实现的开发者

## 3. API 设计

### 3.1 核心接口：Chat Completions

完全兼容 OpenAI Chat Completions API，客户端无需修改代码。

**Endpoint**: `POST /v1/chat/completions`

**请求示例**:
```json
{
  "model": "moa-gpt4-claude",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "解释一下量子计算的基本原理"}
  ],
  "temperature": 0.7,
  "max_tokens": 2000,
  "stream": true
}
```

**响应示例（非流式）**:
```json
{
  "id": "chatcmpl-moa-123456",
  "object": "chat.completion",
  "created": 1699876543,
  "model": "moa-gpt4-claude",
  "choices": [
    {
      "index": 0,
      "message": {
        "role": "assistant",
        "content": "量子计算是一种利用量子力学原理进行计算的技术..."
      },
      "finish_reason": "stop"
    }
  ],
  "usage": {
    "prompt_tokens": 50,
    "completion_tokens": 300,
    "total_tokens": 350
  }
}
```

**响应示例（流式）**:
```
data: {"id":"chatcmpl-moa-123456","object":"chat.completion.chunk","created":1699876543,"model":"moa-gpt4-claude","choices":[{"index":0,"delta":{"role":"assistant","content":"量子"},"finish_reason":null}]}

data: {"id":"chatcmpl-moa-123456","object":"chat.completion.chunk","created":1699876543,"model":"moa-gpt4-claude","choices":[{"index":0,"delta":{"content":"计算"},"finish_reason":null}]}

data: [DONE]
```

```yaml
# MOA 预设配置

      - provider: "anthropic"

        max_tokens: 2000
        timeout: 60
    
    # Aggregator 模型（汇总）

      provider: "openai"

      api_key: "${OPENAI_API_KEY}"
      base_url: "https://api.openai.com/v1"
      temperature: 0.3
      max_tokens: 3000

    # Aggregator Prompt 模板

      1. 保留各回答中的正确信息

        model: "llama3.1-70b"

        max_tokens: 2000
```

### 4.2 配置特性

- **环境变量替换**：支持 `${ENV_VAR}` 语法，敏感信息不硬编码
- **热更新**：配置文件修改后自动重载，无需重启服务

- **灵活 Provider**：支持 OpenAI、Anthropic、本地模型、任何 OpenAI 兼容 API

| **语言** | Python 3.10+ | 生态丰富，LLM SDK 成熟 |

| **配置管理** | PyYAML + Pydantic | 类型安全，校验方便 |
| **部署** | Docker + Docker Compose | 一键部署，环境隔离 |

### 5.2 备选方案：Go + Gin

| 维度 | 选择 | 理由 |
|------|------|------|
| **语言** | Go 1.21+ | 性能极致，部署简单 |
| **Web 框架** | Gin / Fiber | 高性能 HTTP 框架 |
| **LLM 调用** | 自定义 HTTP Client | 直接调用各 Provider API |
| **并发模型** | Goroutine | 轻量级并发，性能优秀 |
| **部署** | 单二进制文件 | 无依赖，部署极简 |

**建议**：第一版用 Python + FastAPI，开发速度快，生态成熟。后续如果性能瓶颈，可以考虑 Go 重写核心引擎。

## 6. 核心模块设计

### 6.1 MOA 编排引擎

```python
class MOAOrchestrator:
    """MOA 编排引擎 - 核心逻辑"""
    
    async def execute(self, request: ChatCompletionRequest, preset: MOAPreset) -> ChatCompletionResponse:
        """执行 MOA 流程"""
        
        # 1. 并行调用所有 Reference 模型
        reference_tasks = [
            self._call_model(ref_config, request.messages)
            for ref_config in preset.references
        ]
        reference_responses = await asyncio.gather(*reference_tasks, return_exceptions=True)
        
        # 2. 过滤失败的响应，记录错误
        successful_responses = []
        for i, resp in enumerate(reference_responses):
            if isinstance(resp, Exception):
                logger.error(f"Reference model {i} failed: {resp}")
                # 可选：降级策略（跳过或使用默认响应）
            else:
                successful_responses.append(resp)
        
        # 3. 构造 Aggregator Prompt
        aggregator_messages = self._build_aggregator_prompt(
            original_messages=request.messages,
            reference_responses=successful_responses,
            template=preset.aggregator_prompt
        )
        
        # 4. 调用 Aggregator 模型
        if request.stream:
            return self._stream_aggregator(preset.aggregator, aggregator_messages)
        else:
            return await self._call_model(preset.aggregator, aggregator_messages)
    
    def _build_aggregator_prompt(self, original_messages, reference_responses, template):
        """构造 Aggregator 的输入"""
        # 将多个 Reference 响应格式化
        formatted_responses = "\n\n".join([
            f"【回答 {i+1}】\n{resp}" 
            for i, resp in enumerate(reference_responses)
        ])
        
        # 填充模板
        prompt = template.format(reference_responses=formatted_responses)
        
        # 构造消息列表
        return [
            {"role": "system", "content": prompt},
            *original_messages  # 保留原始对话历史
        ]
```

### 6.2 模型调用器

```python
class ModelCaller:
    """统一的模型调用接口"""
    
    async def call(self, config: ModelConfig, messages: list, stream: bool = False):
        """调用指定模型"""
        
        # 使用 LiteLLM 统一调用
        response = await litellm.acompletion(
            model=f"{config.provider}/{config.model}",
            messages=messages,
            temperature=config.temperature,
            max_tokens=config.max_tokens,
            api_key=config.api_key,
            base_url=config.base_url,
            stream=stream,
            timeout=config.timeout
        )
        
        if stream:
            return self._handle_stream(response)
        else:
            return response.choices[0].message.content
    
    async def _handle_stream(self, response):
        """处理流式响应"""
        async for chunk in response:
            if chunk.choices[0].delta.content:
                yield chunk.choices[0].delta.content
```

### 6.3 流式处理器

```python
class StreamHandler:
    """流式响应处理"""
    
    async def stream_response(self, aggregator_stream, preset_name):
        """将 Aggregator 流转换为 OpenAI 格式"""
        
        chunk_id = f"chatcmpl-moa-{uuid4().hex[:8]}"
        created = int(time.time())
        
        async for content in aggregator_stream:
            chunk = {
                "id": chunk_id,
                "object": "chat.completion.chunk",
                "created": created,
                "model": preset_name,
                "choices": [{
                    "index": 0,
                    "delta": {"content": content},
                    "finish_reason": None
                }]
            }
            yield f"data: {json.dumps(chunk)}\n\n"
        
        # 发送结束标记
        yield "data: [DONE]\n\n"
```

## 7. 错误处理与降级

### 7.1 错误场景

| 场景 | 处理策略 |
|------|----------|
| 单个 Reference 模型超时 | 跳过该模型，使用其他成功的响应 |
| 所有 Reference 模型失败 | 返回错误，提示所有模型不可用 |
| Aggregator 模型失败 | 返回第一个 Reference 的响应作为降级 |
| API Key 无效 | 返回 401 错误 |
| 速率限制 | 返回 429 错误 |

### 7.2 重试机制

```python
# 指数退避重试
@retry(
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=2, max=10),
    retry=retry_if_exception_type((TimeoutError, ConnectionError))
)
async def call_model_with_retry(config, messages):
    return await model_caller.call(config, messages)
```

## 10. 使用示例

1. 部署 MOA 服务到 `http://your-server:8000`

   - Base URL: `http://your-server:8000/v1`

```python
from openai import OpenAI

client = OpenAI(

)

    model="moa-gpt4-claude",  # 使用配置的 preset name

        {"role": "user", "content": "解释一下量子计算"}
```

## 13. 风险与注意事项

### 13.1 延迟叠加
- **问题**：MOA 本质是两轮调用，总延迟 = max(Reference延迟) + Aggregator延迟
- **缓解**：使用流式输出，让用户尽早看到响应；选择低延迟模型

### 13.2 成本增加
- **问题**：Token 消耗是单模型的 N+1 倍（N 个 Reference + 1 个 Aggregator）
- **缓解**：文档中明确说明；提供低成本预设（本地模型组合）

### 13.3 Aggregator Prompt 设计
- **问题**：汇总效果依赖 Prompt 质量
- **缓解**：提供默认模板；支持自定义；文档中给出最佳实践

### 13.4 模型兼容性
- **问题**：不同模型的输出格式可能不一致
- **缓解**：统一使用 OpenAI 兼容格式；LiteLLM 处理差异
